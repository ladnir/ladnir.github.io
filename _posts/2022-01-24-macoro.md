# C++14/20 Coroutines

The aim of this post is to given another perspective on the C++20 coroutine specification by presenting a small library that implements coroutines using C++14 and a couple macros to aid readability. For example, the following two function <span style="background:#e9e9e9;color:#74531f;">g20</span><span style="background:#e9e9e9;">()</span>, <span style="background:#e9e9e9;color:#74531f;">g14</span><span style="background:#e9e9e9;">()</span> both implementation the same coroutine functionality. While the C++20 version is clearly a bit simpler, both are functionally equivalent and sufficiently readable. Next we will unpack what the ''coroutine transformation'' is and how it we can [partially] implement it without specialized compiler support.

## Coroutine Usage


First, let us review some basic coroutine functionality. If the reader is familiar with coroutines feel free to skip to the next section. Coroutines can be thought of as a stateful function that can "pause" its execution and then be resumed later by the caller or someone else.


###   <span style="background:#e9e9e9;color:#8f08c4;">co_await</span>


There are a couple situations where a coroutine might be pause, or "suspend" in coroutine terminology. The most common is when the coroutine body contains a   <span style="background:#e9e9e9;color:#8f08c4;">co_await</span> call of some [awaitable](https://en.cppreference.com/w/cpp/language/coroutines#co_await) (e.g. some other coroutine) and the result is not ready yet. For example, in the functions above we call <span style="background:#e9e9e9;color:#8f08c4;">co_await</span><span style="background:#e9e9e9;"> h()</span>. The function <span style="background:#e9e9e9;"> h()</span> returns a <span style="background:#e9e9e9;color:#2b91af;">my_future</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:blue;">int</span><span style="background:#e9e9e9;color:#b4b4b4;">&gt;</span> which implements the interface of an "awaitable" with value type <span style="background:#e9e9e9;color:blue;">int</span>. This interface allows the compiler to auto generate some code that determines if the underlying value is ready now or if the caller should suspend while we wait for the value. 


To make the example a bit more concrete, <span style="background:#e9e9e9;"> h()</span> could be fetching a value from some server on the internet which is slow. While we wait, we want to pause this coroutine and allow the caller to go do something else. Onces the value arrives, the coroutine can be resumed by the completion of <span style="background:#e9e9e9;"> h()</span>, on possibly some other thread or we can be notified in some way and manually resume the coroutine. The how/when/who resumes the coroutine can be customized by our return type <span style="background:#e9e9e9;color:#2b91af;">my_future</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:blue;">int</span><span style="background:#e9e9e9;color:#b4b4b4;">&gt;</span> and what we are   <span style="background:#e9e9e9;color:#8f08c4;">co_await</span>ing on.


If the underlying <span style="background:#e9e9e9;color:blue;">int</span> that is provided by <span style="background:#e9e9e9;"> h()</span> is ready immidiately, our coroutine does not need to suspend. 
Whenever the underlying value is ready, either immidiately after the call to <span style="background:#e9e9e9;"> h()</span> or when the coroutine is resumed, execution of the coroutine continues. The next thing to be executed is compiler generated code that passes the value held in the <span style="background:#e9e9e9;color:#2b91af;">my_future</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:blue;">int</span><span style="background:#e9e9e9;color:#b4b4b4;">&gt;</span> to the current coroutine, i.e. assigns it to <span style="background:#e9e9e9;color:blue;">int</span><span style="background:#e9e9e9;"> i</span> in our examples. More details on this compiler generated code later. Finally, execution of the coroutine continues with the next statement after the   <span style="background:#e9e9e9;color:#8f08c4;">co_await</span>.


### <span style="background:#e9e9e9;color:#8f08c4;">co_yield</span>


Another example of when a coroutine can be suspended is with the <span style="background:#e9e9e9;color:#8f08c4;">co_yield</span> keyword. The exact symantics of this is customizable but typically it is used to suspend the current coroutine and pass a value back to the caller. This is similar to returning a value but that we are allowed to resume the coroutine later. Typically, the caller will use this value and then resume to coroutine which can <span style="background:#e9e9e9;color:#8f08c4;">co_yield</span> another value back to the caller or run to completion. A classic example of this is to lazily generate a range of values. For example, <span style="background:#e9e9e9;">range20</span>, <span style="background:#e9e9e9;">range14</span> both return a <span style="background:#e9e9e9;color:#2b91af;">my_range</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:blue;">int</span><span style="background:#e9e9e9;color:#b4b4b4;">&gt;</span> which allows the caller to iterate over the values that are <span style="background:#e9e9e9;color:#8f08c4;">co_yield</span> out of the coroutine. This example prints the string <span style="background:#e9e9e9;">2,3,4,5,6,7,8,9,</span>.

Importantly, the return type of the coroutine, in this case <span style="background:#e9e9e9;color:#2b91af;">my_future</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:blue;">int</span><span style="background:#e9e9e9;color:#b4b4b4;">&gt;</span>, must implement a certain interface in order for the coroutine body to interact with it. In breif, this interface consists of the return type having an associated <span style="background:#e9e9e9;color:#2b91af;">promise_type</span> which implements certain function. For <span style="background:#e9e9e9;color:#8f08c4;">co_yield</span><span style="background:#e9e9e9;"> &lt;expr&gt;;</span> statements the <span style="background:#e9e9e9;">&lt;expr&gt;;</span> value that is yielded is forwarded to <span style="background:#e9e9e9;color:#2b91af;">promise_type</span><span style="background:#e9e9e9;">::yield_value(&lt;expr&gt;)</span> which, in our example, somehow forwards it to the return value <span style="background:#e9e9e9;color:#2b91af;">my_future</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:blue;">int</span><span style="background:#e9e9e9;color:#b4b4b4;">&gt;</span> so that the caller can consume it. Similar "customization points" are required for determining how the other situations are handled.


### <span style="background:#e9e9e9;color:#8f08c4;">co_return</span>
To return a value from a coroutine we must use <span style="background:#e9e9e9;color:#8f08c4;">co_return</span> instead of normal <span style="background:#e9e9e9;color:#8f08c4;">return</span>. This special return keywork helps the compiler know that the current function is a coroutine as opposed to a normal function. In particular, from the function signature <span style="background:#e9e9e9;color:#2b91af;">my_future</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:blue;">int</span><span style="background:#e9e9e9;color:#b4b4b4;">&gt;</span><span style="background:#e9e9e9;color:#74531f;"> g20()</span> there is no way to tell of the body of the function is coroutine or a normal function that just happens to return a <span style="background:#e9e9e9;color:#2b91af;">my_future</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:blue;">int</span><span style="background:#e9e9e9;color:#b4b4b4;">&gt;</span>. 


More generally, whenever the compiler sees at least one   <span style="background:#e9e9e9;color:#8f08c4;">co_await</span>, <span style="background:#e9e9e9;color:#8f08c4;">co_yield</span> or <span style="background:#e9e9e9;color:#8f08c4;">co_return</span> keyword, it will convert the function body into a coroutine by performing the "coroutine transformation." Importantly, this transformation is just an implementation detail. The caller does not know or care if <span style="background:#e9e9e9;color:#2b91af;">my_future</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:blue;">int</span><span style="background:#e9e9e9;color:#b4b4b4;">&gt;</span><span style="background:#e9e9e9;color:#74531f;">  g20()</span> is a coroutine or implements the required logic in some other way.


Next we will discuss percisely that, how to implement the coroutine transformation manually and then introduce the macros you see above to aid readability.


## The Coroutine Transformation


At the core of the coroutine specification relatively mechanical transformation. Every coroutine function is effectively transformed into two functions. The first performs the following:
- Constructs the so-called promise type <span style="background:#e9e9e9;color:#2b91af;">&lt;function-return-type&gt;</span>::<span style="background:#e9e9e9;color:#2b91af;">promise_type</span><span style="background:#e9e9e9;color:#74531f;"> promise</span>. The role of the promise type is to decide when to start and/or resume the coroutine and what to do when it yields a value or returns.
- Construct the return type<span style="background:#e9e9e9;color:#2b91af;">&lt;function-return-type&gt;</span> which in our first example is <span style="background:#e9e9e9;color:#2b91af;">my_future</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:blue;">int</span><span style="background:#e9e9e9;color:#b4b4b4;">&gt;</span>. This is obtained by calling <span style="background:#e9e9e9;color:#74531f;">promise.get_return_object()</span>. This will be what is returned to the caller.
- Query the <span style="background:#e9e9e9;color:#2b91af;">promise_type</span> and decide if the coroutine body should be started now (before returning to the caller) or later (possibly by the caller). This is determined by the result of <span style="background:#e9e9e9;color:#74531f;">promise.initial_suspend()</span>. If we should start now then we call the second function.
- Finally we return the the result of <span style="background:#e9e9e9;color:#74531f;">promise.get_return_object()</span> to the caller.


The second function of the transformation will contain the actual body of the coroutine that the user wrote. Each time   <span style="background:#e9e9e9;color:#8f08c4;">co_await</span>, <span style="background:#e9e9e9;color:#8f08c4;">co_yield</span> or <span style="background:#e9e9e9;color:#8f08c4;">co_return</span> is seen, code in inserted to perform the required logic and allow the coroutine to be suspended/resumed at this location. These are called suspension points.

First the so-called <span style="background:#e9e9e9;color:#2b91af;">coroutine_frame</span> is allocated. The frame will contain the <span style="background:#e9e9e9;color:#2b91af;">promise_type</span> and space for the local variables used in the function body, which for clarity is being managed using <span style="background:#e9e9e9;color:#c8c8c8;">std</span><span style="background:#e9e9e9;color:#b4b4b4;">::</span><span style="background:#e9e9e9;color:#2b91af;">function</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:blue;">void</span><span style="background:#e9e9e9;color:#b4b4b4;">(</span><span style="background:#e9e9e9;color:blue;">void</span><span style="background:#e9e9e9;color:#b4b4b4;">)&gt;</span>. 


This function body can be called many times with the desired logic begining that the function should resume from wherever it left off. The code above acheives this using a hypethetical syntax for storing the addresses of <span style="background:#e9e9e9;color:#8f08c4;">goto</span> labels. In particular, each time we enter the function body we will perform <span style="background:#e9e9e9;color:#8f08c4;">goto</span> <span style="background:#e9e9e9;color:#74531f;">suspend_address</span> where <span style="background:#e9e9e9;color:#74531f;">suspend_address</span> is the location in the function that is just after where we suspended last. 




The first time we enter the function <span style="background:#e9e9e9;color:#74531f;">suspend_address</span> will have the inital value of <span style="background:#e9e9e9;color:#74531f;">InitialSuspend:</span> and begin executing the user code. At some point the execution will hit a <span style="background:#e9e9e9;color:#8f08c4;">co_*</span> statement, in this case <span style="background:#e9e9e9;color:blue;">int </span><span style="background:#e9e9e9;color:#74531f;">i = </span><span style="background:#e9e9e9;color:#8f08c4;">co_await</span><span style="background:#e9e9e9;color:#74531f;"> h()</span>. This statement will be transformed as shown to first query if the value return by <span style="background:#e9e9e9;color:blue;">auto</span><span style="background:#e9e9e9;color:#74531f;">awaiter_h = h()</span> is ready to be consumed. This is acheived by calling <span style="background:#e9e9e9;color:#74531f;">awaiter_h.await_ready()</span>. 


If <span style="background:#e9e9e9;color:#74531f;">awaiter_h.await_ready()</span> returns <span style="background:#e9e9e9;color:blue;">int</span> then we need to suspend the coroutine. This is acheived by updating the <span style="background:#e9e9e9;color:#74531f;">suspend_address</span> to be just after the current   <span style="background:#e9e9e9;color:#8f08c4;">co_await</span> statement and calling <span style="background:#e9e9e9;color:#74531f;">awaiter_h.await_suspend(frame)</span> which gives <span style="background:#e9e9e9;color:#74531f;">awaiter_h</span> the responsibility to resume this coroutine whenever the value does become availiable. Finally we return from the current function, in this case back to the inital caller. Through some other mechanism <span style="background:#e9e9e9;color:#74531f;">awaiter_h</span> will eventially have a value (or exception) and the the function body will be called again. This time the <span style="background:#e9e9e9;color:#8f08c4;">goto </span><span style="background:#e9e9e9;color:#74531f;">suspend_address</span> statement will jump us back to just after the <span style="background:#e9e9e9;color:#8f08c4;">return</span> that was taken last time, i.e. <span style="background:#e9e9e9;color:#74531f;">Suspend_1</span>.


Alternatively, <span style="background:#e9e9e9;color:#74531f;">awaiter_h.await_ready()</span> could have return <span style="background:#e9e9e9;color:blue;">true</span> and the program would similarly be at the <span style="background:#e9e9e9;color:#74531f;">Suspend_1</span> location. Regardless, at this point the result of <span style="background:#e9e9e9;color:#8f08c4;">co_await</span><span style="background:#e9e9e9;color:#74531f;"> h()</span> statement is ready and can be obtained by calling <span style="background:#e9e9e9;color:#74531f;">awaiter_h.await_resume()</span>.


The <span style="background:#e9e9e9;color:#8f08c4;">co_return</span> <span style="background:#e9e9e9;color:#74531f;">i</span> statement is handled by calling <span style="background:#e9e9e9;color:#2b91af;">promise_type</span><span style="background:#e9e9e9;color:#74531f;">return_value(i);</span> and then immidiately proceeding to the so-called <span style="background:#e9e9e9;color:#74531f;">final_suspend</span>. The role of <span style="background:#e9e9e9;color:#74531f;">final_suspend</span> is to determine if the current <span style="background:#e9e9e9;color:#2b91af;">coroutine_frame</span> which contains the user's <span style="background:#e9e9e9;color:#2b91af;">promise_type</span> is safe to be deleted. Often it is prefered to not delete the frame at this time and instead allow the caller/consumer to delete the frame once they have obtained the result, i.e. <span style="background:#e9e9e9;color:#74531f;">i</span> in this case.

The code given above should give you an inutition about what the compiler is doing. However, there are several issues.

### Local variables


The second issue regards the saving the state of local variables between suspend and resume calls. For example, if before <span style="background:#e9e9e9;color:#8f08c4;">co_await</span><span style="background:#e9e9e9;color:#74531f;"> h()</span>  the user has construct some local variable and we suspended for <span style="background:#e9e9e9;"> h()</span>, then when the function is resumed that local variable would be uninitialized.


The compiler handles this in the transformation by moving any declaration of local variables into the <span style="background:#e9e9e9;color:#2b91af;">coroutine_frame</span> as opposed to being the body of our lambda function. They are then destroyed when the coroutine completes.


While one can manually perform the transformation, we will instead consider an alternative solution. In particular, we will place/require all local variables to be declared in the lambda capture. While this results in slightly different symantics, it will enable us to write the <span style="background:#e9e9e9;color:#8a1bff;">CO_BEGIN</span> macro. 


A similar issue regards the lifetime of the awaiter when the coroutine is suspended, i.e. <span style="background:#e9e9e9;color:#74531f;">awaiter_h</span> in the example above. This can be resolved by storing the awaiter either in a fixed sized buffer in <span style="background:#e9e9e9;color:#2b91af;">coroutine_frame</span> or allocating it on the heap if it is too large to fit in the buffer. 


### <span style="background:#e9e9e9;color:#c8c8c8;">std</span><span style="background:#e9e9e9;color:#b4b4b4;">::</span><span style="background:#e9e9e9;color:#2b91af;">coroutine_handle</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:#2b91af;">promise_type</span>&gt;</span>
The final issue regards compatibility with C++20 coroutines. Ideally we can choose to implement our coroutines manually using the pattern outlined here or use the C++20 keywords. However, the primary barrier in achieving is that C++20 defines <span style="background:#e9e9e9;color:#74531f;">&lt;awaiter&gt;.await_suspend</span> as taking a <span style="background:#e9e9e9;color:#c8c8c8;">std</span><span style="background:#e9e9e9;color:#b4b4b4;">::</span><span style="background:#e9e9e9;color:#2b91af;">coroutine_handle</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:#2b91af;">promise_type</span>&gt;</span> as input while our example takes <span style="background:#e9e9e9;color:#2b91af;">&lt;function-name&gt;</span><span style="background:#e9e9e9;color:#b4b4b4;">::</span><span style="background:#e9e9e9;color:#2b91af;">coroutine_frame</span><span style="background:#e9e9e9;color:#74531f;">\*</span>. 


Intuitively, these two parameters represent the same thing. Both are a pointer to the coroutine frame and provide a way for the awaiter to resume this coroutine once the thing being awaited becomes ready. However, the way we resume the <span style="background:#e9e9e9;color:#2b91af;">&lt;function-name&gt;</span><span style="background:#e9e9e9;color:#b4b4b4;">::</span><span style="background:#e9e9e9;color:#2b91af;">coroutine_frame</span><span style="background:#e9e9e9;color:#74531f;">\*</span> vs a C++20 coroutine are concretely different. Moreover, there is no way to convert our <span style="background:#e9e9e9;color:#2b91af;">&lt;function-name&gt;</span><span style="background:#e9e9e9;color:#b4b4b4;">::</span><span style="background:#e9e9e9;color:#2b91af;">coroutine_frame</span><span style="background:#e9e9e9;color:#74531f;">\*</span> into a <span style="background:#e9e9e9;color:#c8c8c8;">std</span><span style="background:#e9e9e9;color:#b4b4b4;">::</span><span style="background:#e9e9e9;color:#2b91af;">coroutine_handle</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:#2b91af;">promise_type</span>&gt;</span> since we can not modify that type. 


This issue could be resolved by giving <span style="background:#e9e9e9;color:#2b91af;">promise_type</span> another customization point which <span style="background:#e9e9e9;color:#c8c8c8;">std</span><span style="background:#e9e9e9;color:#b4b4b4;">::</span><span style="background:#e9e9e9;color:#2b91af;">coroutine_handle</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:#2b91af;">promise_type</span><span style="background:#e9e9e9;color:#b4b4b4;">&gt;</span> interacts with and then dispatches to the correct method for resuming to coroutine. However, this is not part of the standard.


Instead, we will define our own <span style="background:#e9e9e9;color:#c8c8c8;">macoro</span><span style="background:#e9e9e9;color:#b4b4b4;">::</span><span style="background:#e9e9e9;color:#2b91af;">coroutine_handle</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:#2b91af;">promise_type</span>&gt;</span> which can hold either a <span style="background:#e9e9e9;color:#c8c8c8;">std</span><span style="background:#e9e9e9;color:#b4b4b4;">::</span><span style="background:#e9e9e9;color:#2b91af;">coroutine_handle</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:#2b91af;">promise_type</span>&gt;</span> or a <span style="background:#e9e9e9;color:#2b91af;">&lt;function-name&gt;</span><span style="background:#e9e9e9;color:#b4b4b4;">::</span><span style="background:#e9e9e9;color:#2b91af;">coroutine_frame</span><span style="background:#e9e9e9;color:#74531f;">\*</span>. As such, our coroutines will only work with a <span style="background:#e9e9e9;color:#2b91af;">promise_type</span> that can take a <span style="background:#e9e9e9;color:#c8c8c8;">macoro</span><span style="background:#e9e9e9;color:#b4b4b4;">::</span><span style="background:#e9e9e9;color:#2b91af;">coroutine_handle</span><span style="background:#e9e9e9;color:#b4b4b4;">&lt;</span><span style="background:#e9e9e9;color:#2b91af;">*</span>&gt;</span> as input to <span style="background:#e9e9e9;color:#74531f;">&lt;awaiter&gt;.await_suspend(...)</span>.


This is somewhat unfortunent because thirdparty awaitable types which are intended for C++20 will most likely not work out of the box. However this is probable unavoidable given that they will likely take other dependancies to C++20.


### Allocations


The current stratagy requires several allocation. First we must allocate the <span style="background:#e9e9e9;color:#2b91af;">coroutine_frame</span>. A similar stategy can be applied to our framework with some modifcations.

that returns a frame that stores the function body with type <span style="background:#e9e9e9;color:#2b91af;">lambda_type</span> inside the <span style="background:#e9e9e9;color:#2b91af;">coroutine_frame</span>. Then the function body will take a pointer to the frame as its only parameter.


The final allocation is with respect to the storage of any awaiter that are produced during an active   <span style="background:#e9e9e9;color:#8f08c4;">co_await</span>. As suggested previously, these can be partially handled by having an additional buffer within the <span style="background:#e9e9e9;color:#2b91af;">coroutine_frame</span>. However, it is still possible that the coroutine body awaits an awaiter that does not fit in the buffer and then an allocation would occure. However, it is possible for this to result in a compile time error. This can then be combined with allows the buffer size to be an template parameter, i.e. an input to <span style="background:#e9e9e9;color:#8a1bff;">CO_BEGIN</span>. 


Combining these we could obtain a coroutine that is garanteed to not to allocation any memory, something that C++20 coroutines can not currently achieve 100 percent of the time.

## Macros for Readability

Using the pattern above we can define the macros shown in the begining. A prototype of these ideas can be found in the [macoro](http://github.com/ladnir/macoro) repo. Below is a simplified definition of the main macros. In summary <span style="background:#e9e9e9;color:#8a1bff;">CO_BEGIN</span> begins the call to <span style="background:#e9e9e9;color:#74531f;">makeFrame</span> that allocates the frame with the lambda containing the body being stored inline. The closing brace of the lambda/<span style="background:#e9e9e9;color:#74531f;">makeFrame</span> function is in <span style="background:#e9e9e9;color:#8a1bff;">CO_END</span>. The first time we enter the inner function we finish the initial suspend (calling <span style="background:#e9e9e9;color:#74531f;">await_resume</span> and destorying the awaiter). 



<span style="background:#e9e9e9;color:#8f08c4;">co_await</span> first computes the type of the awaiter and then construct it within the frame (or on the heap). It then performs the <span style="background:#e9e9e9;color:#74531f;">await_ready/await_suspend/await_resume</span> proceedure. If it does suspend, it updates the suspend point with the line index.


<span style="background:#e9e9e9;color:#8a1bff;">CO_END</span> first finishes the catch block, passing any exception to the <span style="background:#e9e9e9;color:#2b91af;">promise_type</span> and then performs the final suspend. This completes the definition of the inner function. <span style="background:#e9e9e9;color:#8a1bff;">CO_END</span> then completes the outter function by constructing the return object and starting the initial suspend. If ready we start the coroutine by calling the inner function and then return the return type.
